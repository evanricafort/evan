50+ point JWT security audit checklist based on OWASP, RFC 7519, and real-world pen-testing experience.


    
    

      

        Critical
        Algorithm & Signature Vulnerabilities
      

      

        
🔴
alg:none Bypass
Verify the server rejects tokens with algorithm set to "none". An attacker can forge tokens without a secret.
CRITICAL

        
🔴
Algorithm Confusion (RS256 → HS256)
If the server uses RS256, try sending an HS256-signed token using the public key as the HMAC secret.
CRITICAL

        
🔴
Signature Not Verified
Check whether the server verifies the signature at all. Some implementations only decode without verifying.
CRITICAL

        
🔴
Empty Signature Accepted
Test if the server accepts tokens with an empty/truncated signature field.
CRITICAL

        
🔴
Weak HMAC Secret
Test for short, dictionary-based, or default secrets. Use the Secret Bruteforcer tab to audit.
CRITICAL

        
🟠
Weak Algorithm (HS256)
HS256 may be vulnerable to brute-force if secret entropy is low. Prefer HS512, RS256, or ES256.
HIGH

      

    


    
    

      

        High
        Claims Validation
      

      

        
🔴
Missing exp Claim
Token never expires — remains valid indefinitely after issuance, critical risk if stolen.
CRITICAL

        
🔴
Expired Token Accepted
Verify the server rejects tokens whose exp timestamp has passed. Clock skew should be ≤60s.
HIGH

        
🟠
Missing iss (Issuer) Claim
Without issuer validation, tokens from other systems may be accepted.
MEDIUM

        
🟠
Missing aud (Audience) Claim
Without audience validation, tokens for one service may be used against another (confused deputy).
MEDIUM

        
🟡
Missing jti (JWT ID) Claim
Without a unique ID, token revocation/blacklisting is impossible and replay attacks harder to detect.
LOW

        
🟡
Missing sub (Subject) Claim
The sub claim should uniquely identify the token's subject (usually a user ID).
LOW

        
🟠
Token Not-Before (nbf) Abuse
Verify the server enforces the nbf claim — tokens should not be accepted before the not-before time.
MEDIUM

        
🟠
iat Issued-in-Future
A future-dated iat may indicate token manipulation or clock synchronization issues.
MEDIUM

        
🟡
Excessive Token Lifetime (>24h)
Long-lived tokens increase the impact window if a token is stolen. Prefer short-lived tokens with refresh.
MEDIUM

        
🟡
Replay Attack Potential
Token lacks jti and has a long expiration — replay attacks are harder to detect without a unique ID.
MEDIUM

      

    


    
    

      

        Critical
        Header Injection & Hijacking
      

      

        
🔴
kid SQL Injection
The kid header may be passed to a DB query. Test: kid: 1' OR '1'='1. If key lookup is injectable, attacker can control signature verification.
CRITICAL

        
🔴
kid Path Traversal
If kid is used as a file path, test: kid: ../../dev/null. Allows bypassing signature verification using an empty key.
CRITICAL

        
🔴
kid Command Injection
If kid is passed to a shell command, special characters may lead to RCE.
CRITICAL

        
🔴
jku Header Injection (JWKS URL)
Verify the server validates the jku URL against a trusted whitelist. Malicious jku can redirect key verification to attacker-controlled server.
CRITICAL

        
🔴
x5u Header Injection
Similar to jku — the x5u header specifies a URL for X.509 certificate chain. Must validate against trusted domains only.
CRITICAL

        
🟠
Embedded JWK in Header
Some libraries accept a jwk claim in the header to embed a key. Server should never use an embedded key for verification.
HIGH

        
🟠
x5c Certificate Chain Injection
x5c embeds an X.509 certificate in the header. Validate the certificate chain against trusted roots.
HIGH

      

    


    
    

      

        Medium
        Sensitive Data & Information Disclosure
      

      

        
🟠
Sensitive Data in Payload
JWT payload is base64-encoded, NOT encrypted. Never include passwords, SSNs, credit cards, or PII directly.
HIGH

        
🟡
Excessive Claims / Over-Privilege
Review role/permission claims for over-privilege. Never trust client-supplied privilege escalation.
MEDIUM

        
🟡
Internal System Info in Claims
Avoid leaking internal hostnames, DB names, or architecture details in JWT claims.
LOW

        
🟡
Email / Username Exposed
PII like email or username in JWT can be read by any party that intercepts or stores the token.
LOW

      

    


    
    

      

        Low
        Implementation Best Practices
      

      

        
🟡
Algorithm Whitelist Enforced
Server should maintain an explicit whitelist of allowed algorithms (e.g., only RS256). Never allow "any".
HIGH

        
🟡
Secret Entropy ≥ 256-bit
HMAC secrets must be at least 32 random bytes (256 bits), generated by a CSPRNG.
HIGH

        
🟡
HTTPS Only Token Transmission
Tokens should only be transmitted over HTTPS. Never expose in URLs (query params, paths).
HIGH

        
🟡
Secure Cookie Storage
If using cookies: HttpOnly, Secure, SameSite=Strict flags should all be set to prevent XSS theft.
MEDIUM

        
🟡
Token Revocation / Blacklist
Implement a jti-based token revocation list for logout and compromised credential scenarios.
MEDIUM

        
🟡
Key Rotation Policy
Define and implement a key rotation schedule. Old keys should be retired after rotation.
LOW

        
🟡
Rate Limiting on Token Endpoints
Apply rate limiting to token issuance and verification endpoints to prevent brute-force attacks.
MEDIUM

        
🟡
Unique jti per Token
jti (JWT ID) should use UUID v4 or CSPRNG — not sequential IDs or timestamps alone.
LOW

        
🟡
Clock Skew Tolerance ≤ 60s
Allow at most 60 seconds of clock skew for exp/nbf validation to prevent manipulation.
LOW

        
🟡
Payload Encryption (JWE) for PII
For sensitive payloads, use JSON Web Encryption (JWE / RFC 7516) instead of plain JWT.
MEDIUM
